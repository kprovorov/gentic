# Spec: Direct UI ↔ worker communication over Realtime Broadcast

**Status:** phase 2 being implemented
**Scope:** keep live streaming on Supabase Realtime Broadcast and persist each worker transcript message once, when it finalizes.

## Summary

Today the conversation between the UI and a running agent is routed through the `messages` table: the worker writes throttled row INSERT/UPDATEs via the agent REST API, the browser listens to `postgres_changes`, and user follow-ups are INSERTed by the browser and *polled* by the worker. This spec moves the live conversation onto a **Supabase Realtime private Broadcast channel** per issue (`issue:{id}`). The worker and the browser both connect to the same channel; the database is no longer in the hot path. Durable state (issue claiming, run state, attachments, user messages as run triggers) stays on the existing REST API.

What this buys:

- **True streaming.** Token chunks go straight over a websocket instead of being simulated with throttled row UPDATEs (`StreamingAssistantMessage` in `apps/gentic/src/messages.ts`).
- **Instant follow-ups.** The worker receives user messages as pushed events instead of polling `GET /agent/issues/{id}/messages` every `POLL_INTERVAL_MS` (`nextPrompt` loop in `apps/gentic/src/worker.ts`), which also removes a constant source of agent-API calls and cached Clerk API-key verifications.
- **No per-chunk writes.** No DB rows, no `postgres_changes` fan-out, no Vercel function invocation per transcript update.

What it costs: broadcast has **no replay**. A page refresh mid-run loses the transcript streamed so far (see Consequences). This is accepted for this phase; the follow-up phase ("hybrid") re-adds persistence of *finalized* messages only.

## Current architecture (for reference)

```
agent → UI:   worker ──REST POST/PATCH──▶ web app ──INSERT/UPDATE──▶ messages table
                                                              │
              browser ◀── postgres_changes (issue-chat.tsx) ──┘

UI → agent:   browser ──server action──▶ messages table
              worker  ──REST GET (poll, 3s)──▶ web app ──SELECT──▶ messages table
```

- Worker API surface: `apps/gentic/src/api.ts` (`insertMessage`, `updateMessage`, `fetchUserMessagesAfter`, `setRunState`, `claimNextQueuedIssue`, `fetchAttachments`).
- Streaming simulation: `apps/gentic/src/messages.ts` (`StreamingAssistantMessage`, 250ms flush).
- UI subscription: `apps/web/app/issues/[id]/issue-chat.tsx` (`postgres_changes` on `messages` and `issues`).
- Worker prompt loop: `apps/gentic/src/worker.ts` (`nextPrompt` polls with a cursor; session ends after one quiet poll interval).

## Goals

1. Assistant output (text, thinking, tool events) reaches the browser over a Realtime Broadcast channel with sub-second latency and no DB writes.
2. User follow-up messages reach a live worker session over the same channel, with no polling while a session is active.
3. The worker never holds Supabase credentials more powerful than the user it acts for (it runs on end-user machines — the service key must never leave the web app).
4. Issue claiming, run state, and attachments are unchanged.

## Non-goals

- Persisting the streamed transcript (phase 2, see Future work).
- Replacing `RealtimeRefresh` / board-level `postgres_changes` usage.
- Multi-user collaboration on one issue (design shouldn't preclude it).
- Removing the worker's issue-claim polling loop. Claiming stays REST — the conditional-update claim is a correctness mechanism, not a transport detail.

## Design

### Transport: one private Broadcast channel per issue

- Channel topic: `issue:{issueId}` (issue UUID).
- **Private channels** (`config: { private: true }`), authorized by RLS policies on `realtime.messages`.
- Both the browser and the worker join the same channel; the worker publishes `message` / `run_state` events and subscribes to `user_message`; the browser does the reverse.

### Event protocol

All payloads are JSON. Zod schemas live in a new shared module `@gentic/validators/realtime` so the worker and web agree on one contract.

**Worker → browser**

```jsonc
// event: "message" — full-snapshot upsert of one transcript message
{
  "id": "uuid",              // generated by the worker per message
  "seq": 7,                  // monotonic per message id; receivers drop stale seq
  "role": "assistant",
  "kind": "text" | "thinking" | "tool",
  "content": "full content so far",
  "status": "streaming" | "complete" | "error",
  "ts": "ISO-8601"
}

// event: "run_state" — mirror of the run-state PATCH, for instant UI updates
{
  "status": "cloning" | "in-progress" | "held" | "run-failed" | "ready-for-review" | "waiting-for-input",
  "pr_url": "…" | null,
  "usage_limit_reset_at": "…" | null,
  "run_error": "…" | null,
  "ts": "ISO-8601"
}
```

**Browser → worker**

```jsonc
// event: "user_message"
{
  "id": "uuid",              // the messages-table row id (dual-write, see below)
  "content": "…",
  "created_at": "ISO-8601"
}
```

**Snapshot, not delta.** Each `message` event carries the *full content so far*, throttled (~150ms) exactly like today's `StreamingAssistantMessage` flush. Broadcast delivery is at-most-once; with snapshots a dropped event self-heals on the next flush, whereas a dropped delta would silently corrupt the transcript. `seq` lets receivers discard out-of-order snapshots. This also keeps the UI merge logic identical to the existing `mergeMessage` upsert-by-id.

### Authorization

Three parties, three trust levels:

| Party | Credential | Realtime access |
|---|---|---|
| Browser | Clerk session token (existing `useSupabaseClient`) | RLS-scoped to owned issues |
| Worker | **New:** short-lived Supabase JWT minted by the web app | RLS-scoped to the same user |
| Web app | Service key | unrestricted (not used for this transport) |

**Token exchange.** New endpoint `POST /api/v1/agent/realtime/token` (authenticated like every other agent route via `getAgentContext`):

```jsonc
// response
{
  "url": "https://<project>.supabase.co",
  "apiKey": "<publishable key>",
  "token": "<JWT: sub=<clerk user id>, role=authenticated, exp=now+1h>",
  "expiresAt": "ISO-8601"
}
```

The web app mints the JWT with an **asymmetric private key under Supabase's JWT signing keys feature** (decided — the legacy HS256 shared secret is not used). The private key is a new web-app env var (e.g. `SUPABASE_JWT_PRIVATE_KEY`); Supabase verifies tokens against the corresponding trusted public key. The claims mirror what a Clerk third-party-auth token looks like to RLS (`sub` = Clerk user id, `role = authenticated`), so one set of policies covers both browser and worker. The worker refreshes the token before `expiresAt` and calls `realtime.setAuth(token)` on the live client.

> Implementation note: follow Supabase's current documentation for minting custom tokens with JWT signing keys (key format, registration of the public key with the project) and record the final setup in this doc. Returning `url`/`apiKey` from the endpoint means the worker needs **no new config**; `GENTIC_API_KEY`/`GENTIC_API_URL` remain its only credentials. Neither the service key nor the signing private key may ever reach the worker.

**RLS policies.** New migration adding policies on `realtime.messages`, reusing the ownership join (issues → projects.user_id holds the Clerk id; `issues` has no `user_id` of its own):

```sql
create policy "issue owner can receive broadcasts"
on realtime.messages for select to authenticated
using (
  exists (
    select 1 from public.issues i
    join public.projects p on p.id = i.project_id
    where realtime.topic() = 'issue:' || i.id
      and p.user_id = (select auth.jwt()->>'sub')
  )
);

create policy "issue owner can send broadcasts"
on realtime.messages for insert to authenticated
with check ( /* same predicate */ );
```

### Worker changes (`apps/gentic`)

New module `src/realtime.ts` owning the channel lifecycle:

1. On claiming an issue, call the token endpoint, create a `@supabase/supabase-js` client (new dependency), join `issue:{id}` as a private channel. Channel joined ⇒ proceed; join failure ⇒ fail the run like any other startup error (no silent REST fallback in phase 1).
2. `StreamingAssistantMessage` publishes `message` snapshots to the channel instead of calling `insertMessage`/`updateMessage`. Same throttle structure; the class keeps its shape, only the sink changes. Tool events (`session.ts` `tool_call` branch) publish a single `status: "complete"` message.
3. `setRunState` additionally publishes a `run_state` event after the REST PATCH succeeds (DB stays the source of truth for status; the broadcast is a latency optimization).
4. `nextPrompt` is rebuilt on a push queue:
   - **Seed:** on session start, fetch the backlog once via the existing `GET /agent/issues/{id}/messages?after={cursor}` (messages sent before the worker subscribed — including the issue's initial prompt). The endpoint must start returning `id` alongside `content`/`created_at`.
   - **Live:** `user_message` events append to the queue. Dedupe by `id` against the seeded backlog (a message can arrive via both paths).
   - **End-of-session:** unchanged semantics — after a turn completes, wait one `POLL_INTERVAL_MS` grace for a queued/incoming message, then end.
   - The existing cursor logic for resumed sessions (`runFinishedAt`) applies to the seed fetch only.
5. Token refresh timer; channel teardown in the run's `finally`.

Removed from the worker: `insertMessage`, `updateMessage`, and the in-run polling of `fetchUserMessagesAfter` (kept only as the one-shot seed fetch).

### Web app changes (`apps/web`)

1. **`issue-chat.tsx`:** subscribe to the private broadcast channel (in addition to `postgres_changes` during rollout). `message` events upsert by `id` with the `seq` guard (drop-in for `mergeMessage`); `run_state` events update status/pr_url/usage-limit.
2. **`sendIssueMessage` flow becomes a dual write:** the server action keeps inserting the `messages` row (this is what re-queues a `waiting-for-input` issue and what seeds a later resumed session), and the client *additionally* broadcasts `user_message` on the channel for instant delivery. The row `id` is returned to the client and used as the event `id` so the worker can dedupe.
3. **Agent API:** add the token route; add `id` to the messages GET response. In phase 1, `POST /agent/issues/{id}/messages` and `PATCH /agent/messages/{id}` became unused by the worker but remained temporarily for the phase-2 persistence handoff.

### What stays exactly as-is

- Issue claiming (`POST /agent/issues/claim`) and its poll loop.
- Run state as durable DB state via `PATCH /agent/issues/{id}/run-state`.
- Attachments (REST + signed URLs).
- `RealtimeRefresh` and board-level `postgres_changes` (out of scope).
- Server-rendered initial transcript (`initialMessages`) — phase 2 restores finalized assistant/tool messages while an in-flight message remains broadcast-only until it finalizes.

## Consequences — accepted gaps from phase 1

1. **Refresh mid-run loses the streamed transcript.** The server render shows only persisted rows (user messages); everything the agent said so far is gone from that browser until new events arrive.
2. **Completed runs have no transcript.** History views show user messages, run state, and the PR link only.
3. **At-most-once delivery.** Snapshot semantics contain the damage for `message` events; a lost *final* snapshot leaves a message visually "streaming" until the next event or a refresh. A lost `user_message` event is covered by the DB row + seed fetch on the next run.
4. **Multi-tab:** all tabs subscribe to the channel, so live view works everywhere; only transcript-since-subscribe differs per tab.

## Phase 2 — persist-on-finalize (hybrid)

Streaming stays entirely on broadcast; the worker persists each message **once, when it finalizes** — one INSERT per message via the existing `POST /agent/issues/{id}/messages` route, instead of the pre-realtime INSERT + N UPDATEs. Tool messages insert immediately (they are born complete). User messages are already persisted by the phase-1 dual write. The DB never sees a `streaming` row again.

### Shared ids

The worker already generates a uuid per message for broadcast. The insert endpoint **accepts that id** instead of generating its own, so a message is the same entity on both channels. Everything reconciles from that:

- **Refresh mid-run is lossless.** Server render returns all persisted rows (finalized + user messages); the only thing missing is the in-flight streaming message, and the next snapshot (~150ms, full content) restores it completely via the existing upsert-by-id merge.
- **Double delivery is harmless.** Broadcast event, `postgres_changes` event, and server-rendered row all carry the same id.
- **The phase-1 seed-fetch and dedupe machinery is unchanged.**
- **Idempotent retries.** The endpoint treats a duplicate id as success (return the existing id), so the worker can retry finalize INSERTs blindly.

The `sync_request` event reserved in phase 1 becomes unnecessary — removed from the protocol.

### Ordering

The worker finalizes messages strictly in transcript order, so `created_at` (finalize time) ordering matches the conversation. No started-at column. The only visible artifact: a freshly refreshed page sorts the still-streaming message last, which is where it belongs.

### Failure semantics

1. **Finalize INSERT fails:** the message already reached the user via broadcast, so persistence failure must not kill the run. Retry with backoff; on terminal failure, log and continue — the cost is a gap in history, not a broken session. (Opposite priority from the pre-realtime design, where the DB write *was* the delivery.)
2. **Worker crashes mid-message:** the partial is lost from history — accepted. Refinement: the run's error path best-effort persists the current partial with `status: "error"` so a failed run's transcript shows where it died.

### Scope

- Agent API: `POST /agent/issues/{id}/messages` accepts an optional client-supplied `id` (uuid) with idempotent conflict handling; delete `PATCH /agent/messages/[id]` (unused since phase 1).
- Worker: finalize hook in `StreamingAssistantMessage` inserts the completed message; tool messages insert at emit time; error-path partial persist; remove `updateMessage` from `api.ts`.
- Protocol: drop `sync_request` from `@gentic/validators/realtime`.
- No migration, no auth changes, no UI changes.

This closes phase 1's accepted gaps 1-4: durable transcripts, refresh-safety, completed-run history, and multi-tab convergence.

### Sequencing

Phase 2 lands **before** phase 1's cleanup step (the cleanup would otherwise delete the POST route phase 2 depends on). Final order:

1. Phase 1, rollout steps 1-3 (broadcast transport) — done before this issue.
2. Phase 2 (this issue: persist-on-finalize; deletes only the PATCH route).
3. Cleanup (separate later issue): drop the `postgres_changes` `messages` subscription from `issue-chat.tsx` and replace the `issues` subscription with `run_state` broadcasts. The messages POST route stays — it is now the persistence path.

## Open questions

1. Realtime broadcast rate limits vs. flush interval: at ~150ms throttle a busy run emits ~7 events/s per issue, well under default limits, but confirm plan-level quotas before lowering the throttle further.

## Appendix: JWT signing key setup (implementation record)

The worker token (`POST /api/v1/agent/realtime/token`) is signed with an asymmetric ES256 key registered with the Supabase project as a **standby** JWT signing key — imported, but never rotated into "current" use, so Supabase Auth's own session tokens keep signing with whatever key is already current. A standby key's public half is published on the project's JWKS discovery endpoint as soon as it's imported, which is what makes PostgREST/Realtime trust tokens signed with it; no "Rotate key" step is needed or wanted here.

### 1. Generate the key pair

```bash
supabase gen signing-key --algorithm ES256
```

This prints a private JWK (`kty`, `crv`, `x`, `y`, `d`, `kid`, `alg`).

### 2. Register the public half with the project

In the Supabase dashboard: **Project Settings > API > JWT Signing Keys > Add key > Standby key**, paste the generated JWK. Supabase derives and publishes the public key from it; the `kid` is what our minted JWTs' header must reference. Leave the key as *standby* — do **not** click "Rotate key" or "Migrate JWT secret", either of which would make it the key Supabase Auth uses to sign its own session tokens.

### 3. Configure the web app

Set the full private JWK (the same JSON `supabase gen signing-key` printed) as a single-line JSON string in the web app's environment:

```bash
SUPABASE_JWT_PRIVATE_KEY='{"kty":"EC","crv":"P-256","x":"...","y":"...","d":"...","kid":"...","alg":"ES256"}'
```

`apps/web/lib/realtime-token.ts` imports it with `jose`'s `importJWK`, signs with `alg: ES256` and the JWK's `kid` in the protected header, and sets `sub` = the Clerk user id, `role: authenticated`, `exp` = now + 1 hour — the same claim shape RLS already expects from Clerk-issued tokens (see `20260703150000_projects_user_id_to_clerk.sql`), so the `realtime.messages` policies added in this phase's migration cover both browser and worker tokens with one predicate.

The private key never leaves the web app: the token endpoint returns only the signed JWT plus the public `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`, which is all `apps/gentic/src/realtime.ts` needs to open its own `@supabase/supabase-js` client and join the channel. Neither the Supabase secret key nor this signing private key ever reaches the worker.
